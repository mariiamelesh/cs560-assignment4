<p align="center"><b>Program hierarchy and structure diagram</b></p>  
Program consists of two blocks: cipher and the text editor. They are connected by a cipherApi bridge and a DDL is at a runtime in the TextEditor constructor, and is freed in the destructor. Therefore text editor has no idea about the inside code and implementation of a cipher, it just calls the functions loaded from the lib. Main benefit of such structure is that the system is totally flexible: no matter what cipher we add to the system or change the techniques of ciphering, text editor won't be bothered, since all that class cares about are functions encryptAndSave() and loadAndDecrypt() that are called via API. 

  Inside of the <b>text editor container</b>, there's a Tab structure, which plays as a container for Text class and current state of the tab (History). Consequently, Text is a container for a vector of Lines. Line is an abstract base class for TextLine, Checklist and Contact, allowing us to store various line types inside just one list of lines and keeping the code DRY by passing virtual methods to the derived classes. History serves as a container for storing a list of Copy objects(up to 5 objects) for the undo and redo functions implementation. Copy class contains a serialized version of all the text and the information from Cursor object(cursor position) at some particular moment of time.  

Cipher class is an abstract base class and CaesarCipher is a class derived from the Cipher. Such structure was chosen for the sake of polimorphism which makes it much easier to expand the program in case we would want to add a new cipher type. Inside the CaesarCipher class there's encrypt/decrypt technique implementation. cipherApi.h and cipherApi.cpp serve as a contract by wrapping up the functions implemented in CaesarCipher class into easy-to-export C-functions so they can can be used in DLL.
